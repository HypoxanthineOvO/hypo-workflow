\chapter{引言：从个人 AI 使用经验到工程问题}

\section{问题不是“AI 会不会写代码”，而是“AI 能不能持续做工程”}

这份报告的目标，不是介绍一个新的 prompt 技巧，也不是宣传某个单次对话式开发体验有多惊艳。真正的问题是：当 AI 从“补一段代码”升级为“参与一个跨文件、跨会话、可交付、需要验收和维护的工程过程”时，原有工具往往会迅速暴露结构性缺陷。

作者对 AI Coding 的理解并不是从理论出发，而是从一条非常具体的使用路径出发。最早的阶段主要依赖网页端 GPT / Gemini，或者在 Cherry Studio 中调用 DeepSeek 模型。这一阶段的工具当然能回答编程问题，但无法直接读取完整代码库，也无法稳定持有项目约束。任何一次稍微复杂一点的任务，都要求用户反复补充背景、接口、风格、目录结构和期望行为。表面上是在“问模型”，实际上是在做人肉状态同步。

接下来转向 GitHub Copilot。Copilot 的优势很明确：局部补全很强，续写代码的即时反馈也很好。但它的弱点也同样明确：规划能力有限，对代码库整体约束的感知并不稳定，尤其在多文件协作、跨模块重构和长期风格统一上，仍然需要大量人工干预。这个阶段，作者一度通过堆叠人工 prompt 和外部文档的方式，让 AI 维护 \texttt{CHANGELOG}、\texttt{FEATURES}、版本摘要等文件，并在 Hypo-LaTeX 一类项目上获得初步成效。但这并不意味着问题解决了，只是说明“人可以靠大量手工约束把结果拉回正确轨道”。

随后尝试过 AntiGravity。与更早的工具相比，它的“自动化感”更强，但这种强更多体现在表层交互，而不是工程可靠性上。自动化越多，反而越需要可追踪、可验证、可纠偏的约束系统；否则，错误会被包裹在更漂亮的流程里。AntiGravity 在需求表达空间、行为可控性和验证闭环上的不足，恰好加深了这种认识：自动化本身不是答案，可验证的自动化才是。

寒假开始大规模使用 Claude Code。这是一个真正的拐点。对话式实现能力、上下文组织和中等规模任务的完成度都比前述工具高得多，首次让“Agent 像是在认真协作写代码”这件事变得可信。但新的问题也浮现出来：即使模型足够聪明，用户仍然需要反复粘贴类似的约束，例如输出语言、计划先行、不要越权改文件、修改后要补测试、保持文档同步、按某种风格回答。这说明智力的提升并不自动带来行为固化。

进一步地，作者又接触到 Skills 和类似 Superpowers 的“先计划后做”思路。这一阶段非常关键，因为它第一次把“做之前先拆清楚”正式纳入工作流，而不仅仅是临场提醒。问题在于，这种规划往往只在单次任务里表现优秀；一旦项目跨越多轮迭代，历史背景和前次决策会逐渐从上下文中蒸发，局部最优很容易演化成长期偏航。

再往后，Notion AI 中的 Opus 与向量化检索能力，提供了一个非常强的启发：\emph{规划和检索可以在一个持久知识空间中完成，而具体实现可以交给另一个执行空间。} 从那时起，一个更稳定的协作形态开始成形：用 Notion 类工具承载长期背景、结构化记录和语义检索，用 Codex / Claude Code 负责局部实现和执行。Hypo-Workflow 正是在这条经验链上逐渐凝结出来的。

\section{亲历的工具断裂：从 Hypo-LaTeX 到 Info 的小日记}

如果只把上面的迁移路线写成“工具 A 不够，所以换到工具 B”，这条历史会显得过于干净。真实情况要琐碎得多：每一次工具替换都不是一次漂亮的升级，而是在某个具体项目里被迫承认“这套办法再往下走会把人拖垮”。这些项目本身未必都是 Hypo-Workflow 的直接组成部分，但它们像一组连续实验，把同一个问题从不同角度暴露出来：AI 能在短时间里显得很聪明，却很难在长时间里稳定地承担工程责任。

Hypo-LaTeX 是一个很早的信号。当时作者主要依赖还没有 Skill / project memory 等能力的 Copilot。Copilot 对局部代码、模板续写和小范围修补很有效，但每当任务进入“维护项目秩序”这一层，它就开始显得吃力。例如让它补 \texttt{CHANGELOG}、遵守固定 Git 规范、按既有文档格式整理 release notes、不要乱改不相关文件，这些事情本来应该是项目规则的一部分，却需要作者在每次对话里重新粘贴一大段 prompt。那段时间最强烈的感受不是“模型不会”，而是“模型每次都像刚来这个项目”。它能在当前窗口里理解规则，却无法把规则变成下一次仍然可靠的行为。

更戏剧性的事情是，Copilot 连续被封 3 个账号。这个细节表面上像是账号使用问题，但在方法论上，它逼着作者承认一件事：不能把长期工作流压在单一服务、单一对话入口和单一交互习惯上。一个真正可依赖的工程流程，不能因为某个工具的账号、限流、策略变更或上下文丢失就整体停摆。也正是在这个阶段，“把规则写进外部文件而不是只写进聊天窗口”开始从一个偏好变成一个刚性需求。

\begin{example}{案例：Hypo-LaTeX 的 Prompt 搬运阶段}{case:hypo-latex}
\textbf{项目名：}Hypo-LaTeX。\\
\textbf{当时工具组合：}GitHub Copilot + 手工 prompt + 项目文档。\\
\textbf{做成了什么：}完成了 LaTeX 模板、变更说明、若干规范化文档和重复性维护任务。\\
\textbf{暴露了什么问题：}每次写 \texttt{CHANGELOG}、按 Git 规范提交、保持文档口径一致，都要把同一批规则重新粘贴给模型。Copilot 局部续写很强，但没有稳定的项目级行为记忆。\\
\textbf{促成的转向：}规则必须从 prompt 中移出来，沉淀为 Skill、template、release rule 或其他可复用资产；否则用户只是把“写代码”的负担换成了“重复喂 Prompt”的负担。
\end{example}

Bill 项目则把问题从“规则重复”推进到了“工程失控”。刚开始搭建时，Claude 的表现相当好：它能快速理解需求、生成界面、补齐一些显而易见的状态和交互。但当界面显示开始出问题时，体验迅速变味。作者会指出一个视觉或状态错误，模型给出一个看似合理的修补；修补之后另一个地方又坏掉；继续修，旧问题复发或者新问题出现。更麻烦的是，它并不会自然地把这些修补写成文档、决策记录或回归检查。于是几轮之后，表面上对话里一直在“解决问题”，实际上项目的可理解性越来越差。

这类失败很容易被误判为“模型能力不够”或者“prompt 没写清楚”。但从工程角度看，真正缺失的是全局变更边界、验证策略和文档同步机制。界面问题不是只能靠更聪明的下一次回答解决，它需要明确的验收方式：什么截图算通过，什么状态不允许退化，哪些组件不能被顺手重写，哪些设计约束必须留在文档里。没有这些外部结构，模型越勤快，项目反而越容易被局部修补搅乱。

\begin{example}{案例：Bill 的界面修补泥潭}{case:bill}
\textbf{项目名：}Bill。\\
\textbf{当时工具组合：}Claude + 临时对话式需求说明。\\
\textbf{做成了什么：}前期快速搭建了可运行界面和基础功能。\\
\textbf{暴露了什么问题：}后期 UI / 显示问题进入反复修补状态，模型经常“说一句话干半天”，最终结果却偏离真实需求；文档和回归证据没有同步沉淀，导致每次修复都像从头猜。\\
\textbf{促成的转向：}需要把“用户反馈”转化成明确验收证据，把视觉/交互问题纳入 Test Profile，而不是只靠下一轮自然语言解释。
\end{example}

Agent 项目让 Superpowers 的价值和边界同时变得清楚。Superpowers 前期确实非常好用：它让讨论先落盘，让模型在动手前写计划，让任务看起来不再只是松散聊天。这是作者第一次强烈感受到早期 Harness 的价值。它不是简单 prompt，而是把“先思考、再行动”的节奏外置出来，让模型不至于一上来就乱改文件。对于需求讨论、初始设计、功能分解，它的帮助很明显。

但 Agent 进入持续修 Bug 和加功能之后，Superpowers 的不足也开始暴露。它能在单次任务里形成局部计划，却不擅长维护跨任务的全局观。每次修改看起来都有理由，但理由往往只对当前问题负责；改一个地方，引入一串新的问题；再修新问题，又会冲淡旧设计。最终体验像是模型始终在“为了改而改”，而不是沿着一条可审计的工程轨道推进。这里并不是说 Superpowers 没用，恰恰相反，它的前期有效性证明了 Harness 方向是对的；但它作为早期 Harness 形态，还没有把状态、队列、评估、恢复和长期文档维护真正组合成系统。

\begin{example}{案例：Agent 与 Superpowers 的阶段性成功}{case:agent}
\textbf{项目名：}Agent。\\
\textbf{当时工具组合：}Claude + Superpowers。\\
\textbf{做成了什么：}前期讨论、计划落盘和功能起步效率明显提升。\\
\textbf{暴露了什么问题：}Superpowers 作为早期 Harness 能改善单次任务秩序，却缺少长期全局观；后期修 Bug 和加功能时，局部计划无法可靠约束跨模块影响，常常改一个引入一堆问题。\\
\textbf{促成的转向：}Harness 不能只是一套“动手前先计划”的提示，它必须拥有持久状态、feature queue、step 级检查、report 和 recovery 语义。
\end{example}

Research 和 Info 阶段则暴露出另一种疲劳：作者开始使用 Notion AI 管理文档、沉淀上下文，再把它生成的 prompt 复制给 Codex，让 Codex 执行、总结，再把结果搬回文档。这条路线在能力上比早期工具强很多，因为规划空间和执行空间终于被分开了；但人的角色也变得很尴尬。作者不再主要是“提出工程判断的人”，而更像是在 Notion AI 与 Codex 之间搬运 prompt、搬运结果、搬运上下文。这个阶段的结论非常刺眼：如果规划、状态、检索和执行不能通过协议连起来，那么再聪明的模型组合也会把用户夹在中间当胶水。

\begin{example}{案例：Research / Info 的搬运工阶段}{case:research-info}
\textbf{项目名：}Research / Info。\\
\textbf{当时工具组合：}Notion AI + 文档库 + Codex。\\
\textbf{做成了什么：}长期资料管理、语义检索、报告生成和局部实现能力都明显增强。\\
\textbf{暴露了什么问题：}规划在 Notion，执行在 Codex，状态同步靠作者复制粘贴；模型之间没有共享的生命周期协议，作者本质上变成 Prompt 搬运工。\\
\textbf{促成的转向：}Hypo-Workflow 必须让 plan、state、queue、progress、report、rules 和 execution evidence 都进入同一个文件优先协议，让人不再承担低价值搬运工作。
\end{example}

把这些片段连起来看，Hypo-Workflow 的问题定义就清楚了。它不是为了追赶某一个更强模型，也不是为了把某个工具包装得更好看，而是为了回答一个更朴素的问题：当用户已经知道 AI 可以帮忙写代码之后，如何让这种帮助在真实项目里持续、可恢复、可验证、可解释？从这个角度说，本报告后面的每个功能都不应被理解为“又加了一个模块”，而应被理解为“这个功能解决了怎样的问题”。其中一些能力来自用户反馈，例如对更清晰状态面板、Chat Mode、报告和 Slides 的要求；另一些能力则是作者主动设计出来的办法，例如 Feature Queue、Test Profiles、Progressive Discover 和 file-first recovery 语义。

\section{为什么“多写一点 Prompt”不是长期解法}

很多人第一次意识到 AI Coding 出问题时，会自然地想到一种补救方式：那就把 prompt 写得更详细一些，把规则列得更清楚一些，把上下文补得更全一些。但这种方案的根本问题在于，它把系统性问题转嫁给了人。

如果每次新会话开始，都要重新说明代码库结构、命名约定、交互方式、验收方法、文档要求、输出语言、禁止事项和上次做到哪里，那么所谓“AI 提效”其实只是把开发成本转移成了 prompt 维护成本。对一次性小任务，这种方法可能还凑合；对一个需要持续迭代、需要多人阅读、需要正式汇报的项目，它几乎一定会崩。

更糟的是，prompt 约束即使写得很长，也很难像文件协议一样稳定。它们会被截断、被覆盖、被后续局部细节稀释。换句话说，人类虽然能暂时把模型拉到正确轨道上，但如果轨道本身没有被工程化固化，模型最终仍然会回到“只对当前窗口负责”的局部优化状态。

\begin{important}{核心判断}{intro-judgement}
复杂 AI Coding 的主问题不是单次生成质量，而是长期状态管理、约束固化、验证闭环和中断恢复。
\end{important}

\section{本报告的目标与组织方式}

因此，这份报告试图回答三个问题：

\begin{enumerate}[leftmargin=2em]
  \item 从真实使用经验出发，AI Coding 到底受哪些结构性约束影响？
  \item 如果把这些约束当作工程问题来处理，应当得到怎样的系统设计？
  \item Hypo-Workflow 如何通过一组文件协议、Skills、rules、hooks、reports 和 adapters，把这些设计落地为可执行工作流——并且这些工作流本身是如何从 V4.5 到 C4 逐步演进出来的？
\end{enumerate}

全文按以下逻辑展开：先说明 AI-assisted coding 的核心约束，再提出 Harness Engineering 这一设计视角；随后介绍 Hypo-Workflow 的文件优先协议、分层架构与恢复语义。第二部分以版本时间线为序，从 V4.5 的基础 Pipeline 命令讲起，经过 V5 规划系统、V6 工程化入口、V7 交互增强、V8 修复旁路与上下文管理，再到 C1 OpenCode 平台适配、C2 批量规划与工程自动化、C3 多 Agent 协作与分析预设、C4 知识管理与接受循环——每一条核心命令都从出生场景讲到关键迭代。最后以工具对比、局限性和未来工作收尾。

\chapter{问题定义：AI-assisted coding 的核心约束}

\section{上下文窗口有限，全局观天然不稳定}

当任务涉及多个目录、多文件、历史设计决策和正式交付物时，模型面临的不是“会不会写某一段代码”，而是“能不能持续地维持全局约束”。上下文再长，也只能看到代码库的局部切片，而且会被临近话题覆盖。于是模型经常在局部正确、全局偏航的状态下持续推进。

\section{人工喂 Prompt 的成本会不断累积}

如果用户必须在每个会话里重新说明项目结构、风格要求、输出语言、测试方式和交付格式，那么这套系统本质上要求人来充当外部内存。随着项目时间拉长，这种做法的成本几乎线性增长。

\section{模型会“自己骗自己”}

这是一个很实际的工程问题。模型常常会生成\emph{语气上合理、结构上完整、但事实上不满足约束}的结果。例如：

\begin{itemize}[leftmargin=2em]
  \item 误判某个文件或目录已经存在；
  \item 误以为用户已经确认某种解释；
  \item 只跑单元测试就声称 WebApp 已验收通过；
  \item 在多轮迭代中逐渐偏离既有接口和架构边界。
\end{itemize}

一旦缺少外部检查点，模型就会不断在错误前提上继续推导。

\section{风格与行为难以固化}

很多工具能够“看起来像在记忆项目”，但这种记忆往往只是对当前窗口中显著模式的追随，而不是对长期行为协议的真正继承。没有 Skills、rules、templates、reports 这类外部资产，模型就很难长期稳定地输出同一种工程风格。

\section{中断、恢复和审计困难}

现实开发里，中断是常态。用户会切走去做别的事，会临时插入小修复，会在两个大任务之间聊一些补充问题。如果系统没有明确状态机，就无法回答三个基本问题：

\begin{enumerate}[leftmargin=2em]
  \item 现在做到哪里了？
  \item 下一步该从哪里继续？
  \item 如果最后结果出了问题，问题是在 Discover、Design、Implement 还是 Validation 阶段出现的？
\end{enumerate}

\section{从约束到系统需求}

这些问题不能只停留在体验抱怨层面，而必须转译成系统需求。表~\ref{tab:constraints} 给出了这种映射关系。

\chapter{方法视角：Harness Engineering}

\section{什么是 Harness Engineering}

本项目采用的关键词是 \textbf{Harness Engineering}。这里的 harness 不是模型本体，而是围绕模型构建的一整套外部工程约束层：上下文、计划、状态、规则、评估、恢复与协作协议都被显式写入文件和工具中，让模型始终运行在一条可审计、可恢复、可迭代的工程轨道上。

\section{四个核心原则}

这一定义可以被拆成四个直接的设计原则：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{执行闭环}：Plan、Execute、Review、Report、Evaluate、Recover 组成循环，而不是彼此断裂的聊天片段。
  \item \textbf{分层建模}：Project、Cycle、Feature、Milestone、Patch、Step、Report 各自承担不同粒度的决策。
  \item \textbf{行为固化}：Skills、rules、hooks、templates、adapters 承担“反复重复的约束”。
  \item \textbf{自动维护}：README、PROGRESS、reports、queue、metrics、release checks 由 workflow 自动刷新。
\end{enumerate}

\section{为什么这不是“再造一个 Runner”}

Hypo-Workflow 的关键判断之一是：它不需要自己再造一套执行器。真正执行代码生成与实现工作的，仍然是宿主 Agent，例如 Codex、Claude Code 或 OpenCode。Hypo-Workflow 提供的是围绕这些 Agent 的约束协议和状态系统，而不是替代它们本身。

这一区分很重要。因为一旦 workflow 既想控制状态、又想接管执行、又想提供平台特定 UI，很容易陷入重复发明底层能力的陷阱。Hypo-Workflow 选择的路线是：\emph{统一 contract，多平台 adapter，宿主负责实际推理与执行。}

\section{AI Harness 的设计哲学}

上一节回答了“什么是 Harness Engineering”和“四个核心原则是什么”。本节从更底层的逻辑出发，解释\emph{为什么 Harness 这种方式是有效的}。

\subsection{约束即能力}

传统软件工程的直觉是：约束是限制，限制会降低灵活性。但在 AI-assisted coding 的场景里，这个直觉需要被反转。

模型本身是一个高度灵活的系统：它能生成几乎任何风格的代码，能用几十种方式叙述同一件事，能在一次对话里切换不同的命名约定和架构假设。这种灵活性在短期看起来很美，但在长期工程里会变成毒药：上一轮的实现风格与下一轮不一致，前一个模块的接口约定被后一个模块无意打破，设计意图在迭代中蒸发。

Harness Engineering 的核心假设是：\textbf{对 AI 模型来说，约束不是限制，而是能力的延伸}。当模型运行在一个明确的规则系统内（Skills、rules、templates、hooks），它不需要在每个会话里重新猜测风格、接口、验收标准和恢复策略。这些信息已经从“需要人反复口头重复”变成了“模型可以主动读取的外部轨道”。

换句话说，约束并没有降低模型的“聪明程度”，而是把它的聪明引导到真正需要聪明的地方——设计、实现和问题推理——而不是浪费在反复对齐格式、命名、流程和文档口径上。

\subsection{规划与执行的分离是必然的}

AI Harness 的第二个核心主张是：\textbf{规划和执行必须被显式分开}。

这不是一个偏好问题，而是一个结构性需求。在没有 Harness 的情况下，模型在对话中自然地同时进行规划和执行：它一边理解需求，一边开始写代码。这种方式在短时间内看起来高效，但会导致两个系统性缺陷：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{规划不容易被审计}：如果规划只存在于上一轮对话的上下文里，当上下文被覆盖或对话被中断时，规划的完整逻辑就丢失了。后续的修改缺乏可追溯的设计依据。
  \item \textbf{执行不容易被回滚}：如果执行和规划混在一起，当某个实现步骤出错时，很难判断是规划的问题还是执行的问题。更糟的是，错误的执行结果可能会反向污染规划——\"既然已经写成这样了，那设计就这样吧\"。
\end{enumerate}

Hypo-Workflow 通过 P1-P4 的规划流程和独立的 Step 执行序列，把这两个阶段强制分开。规划的产出（prompts、architecture baseline、test specs）是独立可审查的资产；执行的产出（代码、测试结果、报告）也是独立可验证的资产。当出问题时，可以精确归因到规划的哪个阶段或执行的哪个步骤。

\subsection{文件优先不是技术选择，是哲学选择}

“把状态存进文件而不是聊天窗口”看起来只是一个技术决策，但它背后有一个更深层的主张：\textbf{AI 工程的长期可靠性，依赖的是持久的外部证据，而不是模型的瞬时记忆。}

聊天窗口是短暂的。上下文会被覆盖，会话会结束，模型会被切换。如果把工程状态寄存于这些短暂载体上，项目就永远处于一种“靠人记住和重新说明”的脆弱状态。

文件是持久的。它可以被 Git 追踪、被其他工具阅读、被归档、被跨平台复用。当状态以文件形式存在时，任何 Agent（无论是 Claude、Codex 还是 OpenCode）都可以通过读取同样的文件来理解项目的当前状态，而不需要用户重新叙述。

这就是为什么 Hypo-Workflow 选择 \texttt{.pipeline/} 作为事实来源：不是因为文件天生比聊天窗口好，而是因为\textbf{只有持久的外部证据才能支持长周期的工程可靠性}。

\section{Harness 模式 vs 传统开发模式}

\subsection{传统 AI Coding 模式：对话驱动}

在没有 Harness 的情况下，AI Coding 的典型模式是用对话来驱动一切：

\begin{itemize}[leftmargin=2em]
  \item 用户在聊天中说明需求；
  \item 模型在聊天中生成代码；
  \item 用户在聊天中给出反馈；
  \item 模型在聊天中修改代码。
\end{itemize}

这个模式的根本问题是：\textbf{所有工程状态都埋在对话历史里}。当对话太长，上下文被截断时，早期决策就丢失了。当用户切换到另一个话题再切回来时，模型已经不知道之前做到了哪里。当用户想让另一个模型接手时，对话历史没办法迁移。

\subsection{Harness 模式：文件驱动}

Harness 模式把这个流程改造为文件驱动：

\begin{itemize}[leftmargin=2em]
  \item 需求通过 \texttt{/hw:plan} 的 Discover 阶段写入 prompts；
  \item 规划通过 Decompose 阶段写入 milestone 结构和 architecture 基线；
  \item 执行通过 Step 序列写入 state.yaml 和 reports；
  \item 修复通过 Patch 轨道写入独立的 patch 文件。
\end{itemize}

这个模式的关键优势不在于“做对了什么”，而在于\textbf{能做错之后找回来}。当实现结果不如预期时，可以回到 prompt 看当时怎么规划的；可以回到 report 看哪个 step 出了问题；可以从 state.yaml 恢复中断的执行。工程不是在成功路径上证明自己，而是在失败路径上证明自己。

\section{为什么 AI Harness 是正确的方向}

\subsection{AI 工程需要的不只是更聪明的模型}

过去两年，AI 模型的能力经历了惊人的提升。但一个经常被忽略的事实是：\textbf{模型能力的提升并没有自动解决工程可靠性问题}。更强的模型可以写更复杂的代码、做更深入的分析，但如果缺少约束、规划和恢复机制，这些能力反而会让工程问题变得更隐蔽——错误被包装得更漂亮，偏离被推迟得更远才发现。

Harness Engineering 的基本判断是：\textbf{工程可靠性的提升，依赖的是约束层的厚度，而不是模型智力的高度}。给一个中等能力的模型加上完整的约束系统，它能完成的任务可能超过一个强能力但无约束的模型。

\subsection{Harness 让 AI 从工具变成平台}

如果把 AI 看作“一个能回答编程问题的工具”，那么 Harness 是不必要的——问答确实不需要规划、状态和恢复机制。

但如果把 AI 看作“一个能参与完整工程过程的平台”，那么 Harness 就是必需的——因为真实工程依靠的从来不是天才的一闪念，而是\textbf{可重复的流程、可验证的结果、可追溯的决策、可恢复的状态}。

Hypo-Workflow 不是在做一个更好的 prompt 工具。它是在探索一个方向：当工程本身成为 AI 的“外部骨骼”时，AI 的生产力边界在哪里。

\subsection{从个人实践到系统方法}

本报告第一部分描述了作者从网页 GPT 到 Hypo-Workflow 的个人使用历程。这条路径的深层含义不是“一个用户找到了好用的工具”，而是“一个工程问题找到了它的系统性解法”。

每次工具切换都不是因为新工具“更好用”，而是因为旧工具的约束层太薄，无法支撑更复杂的工程需求。Copilot 的约束层是单文件局部补全；Claude Code 的约束层是单次对话的上下文；Superpowers 的约束层是单次任务的计划框架。Hypo-Workflow 的约束层是\textbf{跨文件、跨会话、跨 Cycle、可归档}的完整工程协议。

这个演进逻辑意味着：AI Coding 的进步不是用更强的模型替换更弱的模型，而是\textbf{用更厚的约束层替换更薄的约束层}。

\begin{important}{AI Harness 核心主张}{ai-harness-thesis}
AI 模型的能力边界，不是由模型本身单独决定的，而是由模型加上它的外部约束系统共同决定的。Harness Engineering 的目标，就是最大化这个约束系统的工程价值。
\end{important}

\end{itemize}

\section{\texttt{/hw:start} — 启动 Pipeline (V4.5)}

\subsection{出生场景}

最早的需求非常朴素：有一个 pipeline 流程，需要一条命令能启动它。当时的假设也很简单——用户总知道自己想做什么，只需要按一下“开始”。

\subsection{设计决策}

\texttt{/hw:start} 的核心设计是“从当前状态开始执行”，而不是每次从头来。它读取 \texttt{.pipeline/state.yaml} 中的当前进度，找到下一个待执行的 Milestone 和 Step，然后启动执行。

\subsection{关键迭代}

在后续版本中，\texttt{/hw:start} 的行为持续完善：
\begin{itemize}[leftmargin=2em]
  \item 支持不同 Step preset（TDD / implement-only / analysis）；
  \item 与 \texttt{/hw:resume} 形成配对，区别在于 resume 只恢复已中断的 pipeline；
  \item C2 后集成 Feature Queue 的 auto-chain 逻辑。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=典型使用场景]
# 开始执行
> /hw:start

# 或从现有状态继续
> /hw:start --resume
\end{lstlisting}

\section{\texttt{/hw:resume} — 恢复中断的 Pipeline (V4.5)}

\subsection{出生场景}

长流程执行总会中断：用户切走、模型出错、上下文爆满。如果每次中断都得从头来，这个系统不可用。

\subsection{设计决策}

\texttt{/hw:resume} 的关键判断是“保持状态可恢复”。它不会覆盖已完成的工作，而是从 \texttt{.pipeline/state.yaml} 中读取最后完成的 step，从下一步继续。与 \texttt{/hw:start} 的区别在于：resume 专门处理“之前已经在执行、但被中断了”的场景。

\subsection{关键迭代}

\begin{itemize}[leftmargin=2em]
  \item C3 引入了 watchdog 自动恢复机制，在检测到 stalled pipeline 时自动触发 resume；
  \item C4 的 Support Bridge 补全了 resume 遇到意外状态文件时的降级逻辑。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=恢复中断的 pipeline]
> /hw:resume
# 从上次中断处继续，不重做已完成工作
\end{lstlisting}

\section{\texttt{/hw:status} — 查看当前状态 (V4.5)}

\subsection{出生场景}

当你有一个能“启动”和“恢复”的系统时，自然会出现下一个问题：现在到底在做什么？\texttt{/hw:status} 就是为这个问题而生的。

\subsection{设计决策}

\texttt{/hw:status} 是只读命令，不修改任何状态。它只读取 \texttt{.pipeline/state.yaml}、\texttt{cycle.yaml} 和 \texttt{PROGRESS.md}，汇总后呈现给用户。

\subsection{关键迭代}

\begin{itemize}[leftmargin=2em]
  \item C2 的 OpenCode TUI 面板将 status 信息可视化到 sidebar/footer；
  \item C4 的全局 Dashboard 为 status 提供 WebUI 展示。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=查看当前进度]
> /hw:status
# 输出：当前 Cycle、Milestone、Step、完成比例、open patches
\end{lstlisting}

\subsection{同一批出厂的命令}

V4.5 还引入了 \texttt{/hw:stop}（停止执行）、\texttt{/hw:skip}（跳过当前步骤）、\texttt{/hw:report}（查看报告）和 \texttt{/hw:config}（查看/修改配置），它们共同构成了基础 Pipeline 操作的最小完备集。

\section{\texttt{/hw:plan} — 四阶段规划 (V5)}

\subsection{出生场景}

V4.5 的命令解决了“能跑”的问题，但随即暴露出一个新问题：在开始执行之前，缺少一个结构化的需求澄清和任务拆分过程。用户常常在还没想清楚要做什么的时候就触发了 \texttt{/hw:start}，结果实现方向偏了。

\subsection{设计决策}

\texttt{/hw:plan} 是 Hypo-Workflow 第一个四阶段命令：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Discover (P1)}：需求澄清、约束收集、仓库分析。
  \item \textbf{Decompose (P2)}：将工作拆分为可审查的 Milestone。
  \item \textbf{Generate (P3)}：生成 prompt、配置、架构基线。
  \item \textbf{Confirm (P4)}：用户最终确认。
\end{enumerate}

这个四阶段结构反映了作者从实战中学到的东西：规划不是一次性活动，而是一个需要多次交互、逐步明确的过程。因此 \texttt{/hw:plan} 默认采用交互式模式（interactive mode），强制要求最少提问轮数。

\subsection{关键迭代}

\begin{itemize}[leftmargin=2em]
  \item V9 (C1)：增加了 Progressive Discover，把 P1 从“问几个问题”升级为从大问题（任务类别、期望效果、验证方法）到小细节（假设、歧义、权衡）的逐步深化；
  \item C2：引入 \texttt{/hw:plan --batch}，支持一次规划多个 Feature；
  \item C3：支持 analysis preset（分析型 workflow）；
  \item C4：\texttt{/hw:plan:review} 支持架构变更后的下游影响分析。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=启动规划流程]
> /hw:plan
# 进入交互式 P1-P4 规划，AI 会逐轮提问

> /hw:plan --batch
# 批量规划多个 Feature
\end{lstlisting}

\section{\texttt{/hw:review} — Plan Review / 架构追踪 (V5)}

\subsection{出生场景}

规划完成之后、执行开始之前，还有一个容易被忽略的需求：确认规划与现有架构之间没有冲突。\texttt{/hw:review} 填补了这个空白。

\subsection{设计决策}

\texttt{/hw:review} 读取 \texttt{.pipeline/architecture.md} 和计划中的 Milestone，检查是否存在架构漂移或不一致。

\subsection{关键迭代}

C4 将 review 升级为 \texttt{/hw:plan:review}，使其成为 Plan 命令族的一部分，并增加了对下游 prompt 影响的分析能力。

\section{\texttt{/hw:init} — 项目初始化 (V6)}

\subsection{出生场景}

V5 有了规划能力之后，下一个自然的问题就是：新项目怎么接入？在此之前，Hypo-Workflow 项目的初始化是手工创建 \texttt{.pipeline/} 目录和相关文件。\texttt{/hw:init} 把这个过程自动化了。

\subsection{关键迭代}

\begin{itemize}[leftmargin=2em]
  \item C1 (V9)：\texttt{/hw:init} 增加了 OpenCode 平台相关的 adapter 生成，包括 \texttt{.opencode/commands/} 和 agent 配置；
  \item C2：增加了 Feature Queue 的初始化支持；
  \item C4：集成 Knowledge Ledger 初始化。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=初始化项目]

# 生成 .pipeline/ 目录和所有必要文件

\section{\texttt{/hw:audit} — 代码审计 (V6)}

\subsection{出生场景}

随着项目规模增长，代码质量问题开始出现：安全漏洞、架构偏离、性能问题。用户需要一种系统化的方式审视代码，而不只是让模型在聊天里“帮忙看看”。

\subsection{关键迭代}

\begin{itemize}[leftmargin=2em]

  \item C4：Knowledge Ledger 让 audit 结果可长期检索和追踪。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=运行审计]
> /hw:audit
# 生成六个维度的分级审计报告
\end{lstlisting}

\section{\texttt{/hw:patch} — Patch 轨道 (V8)}

\subsection{出生场景}

这是 Hypo-Workflow 最深刻的教训之一。C2 执行期间，用户频繁发现小问题（文档拼错、链接失效、表格格式不对），但当时没有轻量修复机制。每个小修复都需要打开一个完整的 Milestone，走 TDD 流程，写 test + implement + review。这个流程对于“修一个错别字”来说太重了。

\subsection{设计决策}

\texttt{/hw:patch} 创建了一条独立的、轻量级的修复轨道。Patch 有自己的状态（open $\rightarrow$ in_progress $\rightarrow$ closed），不强制走 TDD 流程，也不需要创建新的 Cycle 或 Milestone。

设计原则：
\begin{itemize}[leftmargin=2em]
  \item Patch 是轻量的——不需要 Feature、Cycle、TDD 全套流程；
  \item Patch 是独立的——不会阻塞当前 Milestone；
  \item Patch 是可追溯的——每条 Patch 有自己的编号、状态、修复时间和提交。
\end{itemize}

\subsection{关键迭代}

\begin{itemize}[leftmargin=2em]
  \item C3：Patch 增加了与 Agent 矩阵的关联；
  \item C4b：引入了 \texttt{/hw:accept} 和 \texttt{/hw:reject} 的正式验收机制，Patch 可以通过 accept 闭合。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=创建和修复 Patch]
> /hw:patch "修复 README 中的拼写错误"
# 创建一条新 Patch

> /hw:patch fix P001
# 立即修复指定 Patch

> /hw:accept P001
# 验收通过该 Patch (C4b)
\end{lstlisting}

\section{C1 / V9：OpenCode 平台适配}

\subsection{背景：为什么从 Codex 迁移到 OpenCode}

V9 版本的 Motif 来自一个关键的工程判断。V8.4 时，Hypo-Workflow 已经积累了 30 条用户命令、完整的文件协议和 TDD 执行模式——但这一切都是以 Codex / Claude Code 为宿主平台设计的。当作者开始接触 OpenCode 时，发现了几个明显的结构性优势：

\begin{itemize}[leftmargin=2em]
  \item OpenCode 提供了原生 slash command 机制，可以直接将 30 条命令映射为 \texttt{/hw:*} 格式，无需 adapter 做转译；
  \item OpenCode 的 Agent 系统（subagent 绑定）与 Hypo-Workflow 的 Step 委派模型天然对应；
  \item OpenCode 的 event policy 和 file guard 让自动化行为可以被精细控制；
  \item OpenCode 的 TUI 扩展能力提供了在终端内展示 Pipeline 状态的可能。
\end{itemize}

但迁移不是简单地“新增一个 platform adapter”。真正的挑战在于：\emph{如何在不把 Hypo-Workflow 变成 runner 的前提下，提供原生 slash command、agent roles、event policy、file guard 和状态展示。}

\subsection{架构原则}

C1 明确了几个原则，这些原则后来持续影响 C2–C4 的设计：

\begin{enumerate}[leftmargin=2em]
  \item 一套 workflow contract，多平台 adapter；
  \item OpenCode 原生能力优先，不重复发明执行机制；
  \item deterministic generation 进入 \texttt{core/}；
  \item 实际推理与实现仍由宿主 Agent 完成；
  \item \texttt{.pipeline/} 保持跨平台 source of truth。
\end{enumerate}

\subsection{核心工程判断}

C1 共完成 10 个 Milestone，产出包括 OpenCode native adapter、global CLI、30 条 slash command 映射、plugin scaffold、TUI 扩展基础、文档和 release 准备。几个关键判断：

\begin{enumerate}[leftmargin=2em]
  \item \texttt{core/} 只承载 deterministic helper，不承担 runner 语义——这是 C1 最核心的架构决策，避免了“workflow 变成 runner”的陷阱；
  \item 命令映射必须可测试——C1 为每一条 slash command 编写了回归测试，确保映射不出错；
  \item plugin 与 runtime helper 分离——避免被 OpenCode 错误扫描和注入；
  \item regression suite 不是装饰品，而是交付证据的一部分——C1 的 10 个 Milestone 每个都有对应的测试报告。
\end{enumerate}

\subsection{为什么 C1 说明这套方法有效}

C1 的价值不只是“支持了 OpenCode”，而是它把一组原本容易散落在聊天记录中的设计约束，落到了可验证资产中：capability matrix、platform adapter、Ask/todowrite 纪律、file guard、bootstrap artifact、文档和 release metadata 都进入了工程系统。

\subsection{版本体系切换}

C1 也是 Hypo-Workflow 从数字版本命名（V8.4 $\rightarrow$ V9.0）切换到 Cycle 命名（C1）的转折点。这一变化反映了工程观念上的一个转变：不再把每个版本当作孤立发布，而是当作一个“交付边界”——有明确的摘要、经验教训、归档和可恢复性。

\section{\texttt{/hw:plan --batch} — Feature Queue 批量规划 (C2)}

\subsection{出生场景}

当 Hypo-Workflow 的单个 Feature 规划流程成熟后，很快就遇到了新的瓶颈：C2 是一个大型 Cycle，需要规划 20 个 Milestone、跨越多个 Feature。如果每个 Feature 都需要单独的 \texttt{/hw:plan} 会话，规划本身就会成为一个时间瓶颈。

更关键的是，Feature 之间有依赖关系和优先级，需要一个调度层来管理。

\subsection{设计决策}

\texttt{/hw:plan --batch} 引入了一个全新的抽象层——\textbf{Feature Queue}。用户可以在一次规划中列出多个 Feature，系统会：

\begin{itemize}[leftmargin=2em]
  \item 为所有 Feature 运行一次统一的 Discover 访谈；
  \item 生成 \texttt{.pipeline/feature-queue.yaml}，包含所有 Feature 的优先级、依赖和 gate；
  \item 根据 \texttt{batch.decompose\_mode}（upfront / just\_in\_time）分解 Milestone；
  \item 生成 Feature 级 Markdown 表格和 Mermaid 依赖图；
  \item 在 Batch 执行时自动链式推进（auto\_chain）。
\end{itemize}

\subsection{关键迭代}

C2 还引入了两个与 batch plan 密切相关的机制：

\begin{itemize}[leftmargin=2em]
  \item \textbf{Progressive Discover}：把 P1 从“问几个问题”升级为从大问题到小细节的逐步深化。这对 batch plan 特别重要，因为在一次访谈中需要收集多个 Feature 的信息，结构化的提问策略有助于避免遗漏。
  \item \textbf{Test Profiles}：为不同类别的工作定义不同的验收标准。WebApp 需要 E2E + browser interaction 证据，agent-service 需要 CLI plan + 真实执行，research 需要 baseline + 前后对比。这对于 batch plan 中的多个 Feature 各需要不同验证方式特别有用。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=批量规划]
> /hw:plan --batch
# 进入批量 Discover，一次为多个 Feature 收集需求和约束

> /hw:plan --insert "在 F003 和 F004 之间插入 F3.5"
# 向已有的 Feature Queue 插入新 Feature
\end{lstlisting}

\section{\texttt{/hw:chat} — 追加对话模式 (C3)}

\subsection{出生场景}

这是 C3 需求页上明确提出来的需求。在正常执行过程中，用户常常会产生一些“追加的问题”——想继续讨论、想做一个小的追改、想确认一个细节，但不想打开一个新的 Milestone 或 Patch。

\subsection{设计决策}

\texttt{/hw:chat} 是一个轻量级的追加对话模式。它与完整 Milestone 的区别在于：

\begin{itemize}[leftmargin=2em]
  \item 不创建新的 Milestone；
  \item 不修改 \texttt{state.yaml} 中的执行进度；
  \item 对话内容追加到日志中，保持可追溯。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=进入轻量讨论模式]
> /hw:chat
# 进入追加对话，不创建新 Milestone
# 适合：补充讨论、小追问、确认设计细节
\end{lstlisting}

\subsection{C3 同时期的其他产出}

C3 的另一个核心贡献是 \textbf{Agent 矩阵}——定义了 7 个专用 subagent（hw-code-a, hw-code-b, hw-explore, hw-review, hw-test, hw-docs, hw-debug），让不同性质的子任务可以分配到不同 Agent。同时，C3 完整实现了 \textbf{analysis preset}：为审计、调试、根因分析类工作定义了标准化的 analysis workflow（define\_question $\rightarrow$ gather\_context $\rightarrow$ hypothesize $\rightarrow$ experiment $\rightarrow$ interpret $\rightarrow$ conclude）。

\section{\texttt{/hw:accept} / \texttt{/hw:reject} — Cycle/Patch 验收 (C4b)}

\subsection{出生场景}

在 C4 之前，Cycle 和 Patch 的“完成”是一个软概念。一个 Patch 标示为 closed，一个 Cycle 标示为 done，但缺少用户正式认定的“验收通过”或“打回”动作。

\subsection{设计决策}

\texttt{/hw:accept} 和 \texttt{/hw:reject} 是 C4b 引入的验收闭环：

\begin{itemize}[leftmargin=2em]
  \item \texttt{/hw:accept}：标记 Cycle 或 Patch 验收通过，更新状态并触发后续动作（如释放 feature queue 中的下一个 Feature）；
  \item \texttt{/hw:reject}：打回 Cycle 或 Patch，附带原因，允许重新修改后再次提交验收。
\end{itemize}

\subsection{关键迭代}

Acceptance 与 Feature Queue 的 auto\_chain 紧密集成：一个 Feature 的所有 Milestone 完成并通过 accept 后，系统自动推进到下一个 Feature（当 auto\_chain=true 时），除非下个 Feature 的 gate 设为 confirm。

\subsection{用法}

\begin{lstlisting}[caption=验收 Cycle 或 Patch]
> /hw:accept C4
# 验收通过 C4 Cycle

> /hw:reject P005 --reason "需要补充回归测试"
# 打回 Patch，附带原因
\end{lstlisting}

\section{\texttt{/hw:explore} — 探索模式 (C4c)}

\subsection{出生场景}

AI 拿到一个新项目时，最大的挑战不是“不会写”，而是“不知道从哪里开始了解”。用户想要一种结构化的方式，让 AI 帮助探索未知项目，产出可供后续 Plan 使用的上下文。

\subsection{设计决策}

\texttt{/hw:explore} 是一种受控的探索模式：

\begin{itemize}[leftmargin=2em]
  \item 在独立的 Worktree 中运行，不污染主项目状态；
  \item 产出探索报告：项目结构、关键文件、依赖图谱、潜在入口点；
  \item 探索结果可作为 \texttt{/hw:plan} 的上下文输入（\texttt{--context explore}）。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=探索新项目]
> /hw:explore
# 触发探索模式，产出结构化探索报告

> /hw:plan --context explore
# 将探索结果作为规划上下文
\end{lstlisting}

\section{\texttt{/hw:knowledge} — 知识查看 (C4d)}

\subsection{出生场景}

长周期项目中，AI 和用户都会忘记一些东西。历史决策、技术约束、架构约定——这些都散落在不同的 prompt、报告和日志文件中。\texttt{/hw:knowledge} 的目标是让这些知识可检索、可回顾。

\subsection{设计决策}

\texttt{/hw:knowledge} 是 Knowledge Ledger 的用户接口。它：

\begin{itemize}[leftmargin=2em]
  \item 从 \texttt{.pipeline/knowledge/} 读取项目知识条目；
  \item 支持按标签、类别、时间范围过滤；
  \item 受 session scope 限制——当前会话只能加载相关范围的知识（避免上下文膨胀）。
\end{itemize}

\subsection{用法}

\begin{lstlisting}[caption=查看项目知识]
> /hw:knowledge
# 显示当前项目知识库摘要

> /hw:knowledge --scope session
# 只加载当前会话相关的知识
\end{lstlisting}

\section{\texttt{/hw:sync} — 显式状态同步 (C4d)}

\subsection{出生场景}

Hypo-Workflow 的文件协议虽然让状态持久化了，但多个并发会话、多 Worktree、跨平台协作时，状态同步仍然是一个挑战。

\subsection{设计决策}

\texttt{/hw:sync} 提供显式的项目状态同步，确保多个入口看到的状态是一致的。

\subsection{用法}

\begin{lstlisting}[caption=同步项目状态]
> /hw:sync
# 显式触发状态同步，确保多入口一致性
\end{lstlisting}

\section{对 Copilot 和 AntiGravity 的技术性批评}

本报告第 1 章从个人使用经验出发提到了一些工具的具体问题，这里从技术角度进行更系统的分析。

Copilot 的核心设计是“补全当前行/当前文件”。这个思路在局部补全场景下效率很高，但在工程层面有三个结构性问题：

\begin{enumerate}[leftmargin=2em]
  \item 它没有跨文件的持久状态概念；
  \item 它没有规划、验收和恢复语义；
  \item 它依赖 IDE 内置上下文，但无法在长对话中保留远距离约束。
\end{enumerate}

AntiGravity 的问题则更微妙。它的自动化程度更高，交互体验更流畅——但这些优势恰恰放大了缺少约束机制的风险。自动化越强，越需要可验证的约束系统。否则，“一次漂亮的操作”和“一次隐蔽的错误”可能会以同样的方式呈现。

\section{从个人工具迁移到系统设计}

回顾从网页 GPT 到 Hypo-Workflow 的整条路径，可以提炼出一个更宏观的判断：

\begin{important}{演进判断}{evolution-judgement}
AI Coding 工具的进化不是“更聪明的模型替代更老的模型”，而是“更完整的约束层替代更薄弱的约束层”。每个阶段的进步，都来自约束机制的显式化和工程化。
\end{important}

这一判断也解释了为什么 Hypo-Workflow 选择了“不做 Runner”的路线：因为 Runner 本身就是一种约束形式（让行为可重复、可调度），但 Runner 的约束力不如文件协议的约束力稳定和持久。

\chapter{局限性与未来工作}

\section{局限性}

作为一个工程尝试，Hypo-Workflow 在当前阶段仍有一些明确的局限：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{学习成本}：P1-P4 规划和 TDD 执行模式需要用户理解文件协议、Cycle 语义和状态模型。对于“只想快速问一个问题”的用户，这套机制显得过重。
  \item \textbf{平台耦合}：虽然设计上“统一 contract，多平台 adapter”，但目前主要适配平台仍是 OpenCode。Codex、Claude Code、VS Code Agent 等平台的 adapter 需要更多验证。
  \item \textbf{上下文占用}：.pipeline/ 中的文件会占用 Agent 上下文。虽然 compact 视图缓解了这个问题，但在极大型项目上仍不够。
  \item \textbf{没有分布式协作}：当前设计假设一个 Agent 在同一时间处理一个项目。多用户/多 Agent 并发协作需要更复杂的协议。
  \item \textbf{模型工程智力需求}：Hypo-Workflow 当前仍然依赖较强的模型能力（规划、推理、多步执行）。一个值得探讨的问题是：Harness Engineering 能否降低对模型工程智力的需求，让中等能力的模型也能通过外部约束完成复杂工程任务？这一方向尚未充分探索。
\end{enumerate}

\section{未来工作}

基于上述局限性和当前的技术趋势，以下几个方向值得进一步探索：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{多模型 Harness 对比}：使用相同的约束协议，对比不同模型在 Hypo-Workflow 上的工程表现。目标是验证 Harness 是否能让中等模型也能做到当前强模型才能做的事。
  \item \textbf{跨平台 adapter 矩阵}：补齐 Codex、Claude Code、VS Code Agent 等平台的 adapter，形成统一的 adapter 测试矩阵。
  \item \textbf{Knowledge Ledger 扩展}：将知识从当前项目的 structured 记录扩展到跨项目的模式识别和迁移学习。
  \item \textbf{验收自动化}：探索 \texttt{/hw:accept} 和 \texttt{/hw:reject} 的部分自动化——当所有检查（回归、代码质量、匹配计划）通过时，自动确认验收。
  \item \textbf{探索模式与自主执行}：当前 explore 模式以受控探索为主。随着对模型行为的信任提升，可以探索“自主探索 $\rightarrow$ 自动生成 Plan $\rightarrow$ 带确认执行”的链路。
\end{enumerate}
