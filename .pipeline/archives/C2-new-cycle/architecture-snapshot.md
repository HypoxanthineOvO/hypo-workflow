# Architecture Baseline — C2 Maintainability, Observability, Batch Plan, and Showcase

## Current Baseline

- Active Cycle: C2.
- Previous Cycle C1 delivered V9 OpenCode Native Adapter and is archived under `.pipeline/archives/C1-v9-opencode-native-adapter/`.
- Current active `.pipeline/prompts/` is empty after archive, so C2 can generate a fresh 12-milestone sequence.
- `.pipeline/` remains the source of truth for Cycle, state, rules, progress, logs, patches, prompts, reports, and generated planning artifacts.
- Protected files remain `.pipeline/state.yaml`, `.pipeline/cycle.yaml`, and `.pipeline/rules.yaml`; implementation prompts should not mutate them outside expected workflow state transitions.

## C2 Architecture Direction

C2 adds five Feature groups:

1. README automation integrated into release.
2. Skill system quality contract and cleanup.
3. Batch planning through a persistent Feature Queue.
4. OpenCode TUI status panels backed by `.pipeline/` files.
5. A technical report and Beamer slides that explain the project.

## Core Concepts

### Hierarchy

- Project contains Cycles.
- Cycle contains Features.
- Feature contains Milestones.
- Milestone remains the executable prompt/report unit.
- Patch is a cross-Cycle side track with `discovered_in` and `resolved_by` references.
- Milestone numbering stays Cycle-global; `feature_id` records ownership.

### File Protocol

- `.pipeline/config.yaml`: project-level planning/execution overrides.
- `.pipeline/design-spec.md`: human-readable C2 requirements and design intent.
- `.pipeline/prompts/*.md`: executable Milestone prompts generated by P3.
- `.pipeline/feature-queue.yaml`: future batch queue state and scheduling source.
- `.pipeline/metrics.yaml`: future detailed telemetry by Cycle, Feature, Milestone, and Step.
- `.pipeline/log.yaml` and `.pipeline/PROGRESS.md`: lifecycle history and progress.
- `.pipeline/reports/*.md`: Milestone outcome reports.

### Release and README

README dynamic content should be regenerated from project assets through a controlled update path:

- `templates/readme-spec.md` defines structure and data sources.
- Managed marker blocks are the default replacement boundary.
- Full README regeneration is profile/config-gated.
- `/hw:release` gains an `update_readme` substep before commit/tag.
- `readme-freshness` becomes a pre-release rule/check.

### Skill System

Skill cleanup is conservative:

- No merge/delete in C2.
- `references/skill-spec.md` becomes the quality contract.
- `skill-quality` catches structural issues without silently changing command semantics.
- Codex, Claude Code, and OpenCode mappings must remain traceable.

### Batch Plan

Batch Plan adds a Feature-level queue above Milestones:

- `/hw:plan` remains unchanged for single-feature planning.
- `/hw:plan --batch` performs multi-feature Discover and writes a prioritized queue.
- Default `batch.decompose_mode` is `upfront`; `just_in_time` is supported.
- `/hw:plan --insert` edits the queue through natural-language operations and confirmation gates.
- Auto-chain advances Feature by Feature, with `gate: confirm` as a pause point.

### OpenCode TUI

OpenCode UI reads status from `.pipeline/` files:

- Sidebar: Cycle, Feature Queue, current Feature, Milestones, blocked/deferred state, recent events, duration/token/cost summary.
- Footer: current step, progress, latest evaluation score, heartbeat, Ask gate/failure state, latest event.
- UI logic is adapter behavior, not a separate runner.
- Token/cost is best-effort; unavailable values render as `n/a`.

### Report and Slides

The report/slides explain Hypo-Workflow as a Harness Engineering project:

- Mainline: personal AI tool experience leads into system design.
- Core axes: execution loop, layering, behavior solidification, automatic maintenance.
- V9 OpenCode Native Adapter is the detailed Case Study.
- Slides use a classroom technical-report style, not a business pitch.

## Review Expectations

- After each Milestone, run `/hw:plan:review` if architecture, schemas, or command semantics changed.
- Do not let README automation, batch mode, or OpenCode UI regress existing single-feature `/hw:plan`.
- Final validation must cover core tests, README freshness, Skill quality, batch queue behavior, OpenCode artifact generation, and report/slides compilation where tools are available.
